# Vue 3 前端开发规范

## 1. 适用范围

本文档适用于使用 Vue 3 开发、并与 [Python 后端开发规范](python-backend-development-guide.md) 中所述 FastAPI 服务进行接口联调的前端项目。

当前已确定前端框架为 `Vue 3`，开发语言使用 `TypeScript`。构建工具、包管理器、路由库、状态管理库、请求库和 UI 组件库尚未指定，不在本文档中擅自固定。

## 2. 已确定约束

| 项目 | 要求 | 说明 |
| --- | --- | --- |
| 前端框架 | `Vue 3` | 页面和组件基于 Vue 3 开发 |
| 开发语言 | `TypeScript` | Vue 页面、组件和共享逻辑使用 TypeScript 编写 |
| 后端接口 | `FastAPI` | 前端与 Python 3.12 / FastAPI 后端进行联调 |
| 数据服务兼容背景 | PostgreSQL `12`、Redis `6.2.5` | 前端不直接访问服务，但应遵循后端提供的数据契约 |
| 服务启动背景 | `Uvicorn` | 本地调试连接实际 FastAPI 服务地址；打包运行时使用浏览器同源接口入口 |
| 共享功能一致性 | 必须保持一致 | 多页面使用同一功能时遵循 [跨页面功能一致性约定](cross-page-feature-consistency.md) |

## 3. Vue 3 开发约定

### 3.1 页面与组件职责

- 页面组件负责页面级布局、路由参数接入和业务流程编排。
- 可复用展示或交互能力应沉淀为公共组件，不在多个页面复制相近实现。
- 与视图无关、可复用的业务处理逻辑，应从页面组件中抽离为共享逻辑模块或组合式函数；具体组织目录在项目结构确定后补充。
- 组件对外输入、输出和状态变化应清晰，避免页面依赖组件内部隐含行为。
- 当组件以列表形式展示数据时，列表标题与列表内容应居中展示，并保持标题列与内容项的对齐口径一致。
- 当组件以树形结构展示上下级级联数据时，应支持点击整行展开或收起节点，不仅依赖单独的展开图标。

### 3.2 状态与副作用

- 状态应放在最接近使用范围的位置；仅在多个页面或跨组件确实共享时使用共享状态方案。
- 接口请求、定时任务、订阅或缓存读取等副作用，应有明确的触发时机与清理方式。
- 不将同一业务数据在多个页面分别维护为互相独立且可能冲突的状态来源。
- 树形结构的展开状态应作为明确的页面或组件状态维护；刷新数据、编辑、修改、新增或删除后，应按稳定节点标识恢复此前的展开或收起布局，避免无关节点被重置。

### 3.3 复用与一致性

- 同一个业务功能被不同页面使用时，应复用统一组件、共享逻辑和接口调用规则。
- 同一字段在不同页面应保持相同含义、格式化方式和校验口径，除非用户确认存在特殊场景。
- 如确实需要针对某页面实现不同表现，应在实施前告知用户差异原因、受影响页面和与统一方案的不同之处。

### 3.4 TypeScript 类型约定

- Vue 组件的输入、输出以及共享逻辑的外部接口应使用明确的 TypeScript 类型表达。
- API 请求参数和响应数据应定义类型，并与 FastAPI 接口已确认的数据结构保持一致。
- 不通过滥用宽泛类型绕开接口或组件边界问题；确需例外时，应说明原因和影响范围。
- 对接口中的可空字段、枚举值、时间值、分页参数和标识字段，应依据后端契约表达类型，不凭页面展示结果猜测类型。
- 类型定义在多个页面共用时，应集中维护并复用，不为相同响应结构分别声明含义不一致的类型。

## 4. 与 Python/FastAPI 后端协作

### 4.1 API 契约

- 前端只依赖后端已提供或已确认的 API 路径、请求参数、响应结构和错误语义，不自行假设字段。
- 接口的数据校验和业务规则以 FastAPI 后端确认的契约为基础；前端校验用于提升交互体验，不替代后端校验。
- TypeScript 中的 API 请求和响应类型应随已确认的 FastAPI 模型变化同步更新。
- 后端接口契约发生变更时，检查所有调用该接口的 Vue 页面与共享模块，保持多入口表现一致。
- 后端尚未确认统一响应封装、分页结构、鉴权方式或错误码规范前，前端不得将某种格式作为全项目固定约束。

### 4.2 数据归属与字段使用

- PostgreSQL 和 Redis 均由 Python 后端负责访问；Vue 前端不得直接连接数据库或 Redis。
- 列表筛选、排序、分页、搜索和权限过滤等会影响业务结果的数据规则，以 FastAPI 接口能力和后端结果为准。
- 前端仅负责展示格式化、输入交互校验和页面状态管理，不在页面中重新实现会改变业务结果的数据计算规则。
- 对时间、金额、状态枚举、空值和标识字段的展示处理，应基于后端已确认字段语义；语义不明确时先确认再实现。

### 4.3 接口地址配置

- 打包构建后的前端运行时，请求后端接口地址应使用浏览器当前访问的协议、主机和端口作为基础地址，仅拼接 API 路径。
- 打包运行时不得额外写死或拼接独立后端主机、端口；该模式要求后端接口通过浏览器当前访问地址的同源入口可访问。
- API 路径前缀应通过统一配置或统一请求层维护，不分散写在页面组件中。
- 本地开发调试时，应支持直接配置后端接口完整地址，以便连接本地或指定调试环境中的 FastAPI 服务。
- 开发调试地址和打包运行时地址解析规则应区分清楚，避免将本地固定地址带入生产构建。

### 4.4 请求封装与错误处理

- API 调用应采用统一封装方式，集中处理基础地址、鉴权信息、通用错误和响应解析；具体请求库待确认。
- API 基础地址应由统一请求层按运行模式解析，不在组件或页面中重复硬编码后端地址。
- 同一接口在多个页面被调用时，应复用请求函数和对应 TypeScript 类型定义。
- 页面应提供加载中、成功、无数据和失败等必要状态反馈，避免接口失败后页面无可解释反馈。
- 对后端返回的业务错误，应使用一致的展示和处理规则；特殊页面行为需经确认后记录。
- 前端不得把后端校验失败、权限失败或业务冲突静默转换为成功状态。

### 4.5 Redis 相关数据一致性

- 当页面展示内容来自后端缓存、会话或限流等 Redis 相关能力时，前端只依据接口响应展示状态，不假设 Redis 内部键结构或过期策略。
- 涉及更新后立即刷新列表、详情或状态的功能，应与后端确认写入后读取的一致性表现；前端不以自行缓存掩盖接口结果差异。
- 若同一数据在多个页面展示，更新后的刷新或失效策略应统一；需要页面特殊处理时，依照 [跨页面功能一致性约定](cross-page-feature-consistency.md) 告知用户。

### 4.6 联调与验证

- 开发需要接口数据的页面前，确认 FastAPI 接口定义、字段类型和错误响应规则。
- 本地调试时验证显式配置的后端接口地址可正确访问 FastAPI 服务；打包验证时检查请求地址使用浏览器当前协议、主机和端口，仅追加 API 路径。
- 联调时覆盖正常响应、空数据、校验失败、权限或业务错误等与功能相关的场景。
- 当页面功能涉及后端至少 `50` 并发请求的容量目标时，前端不应产生无必要的重复请求或高频轮询；具体性能验收依据后续场景确定。
- 搜索、筛选或联想输入触发请求时，应避免每次无控制地重复访问后端；具体防抖、取消或合并方式按实际交互确定。
- 提交、保存、删除等写操作应避免因重复点击产生并发重复请求，并根据后端确认结果更新页面。

## 5. 目录与职责边界

- 页面层：负责路由页面组合、数据展示和用户交互流程。
- 组件层：负责可复用界面与交互单元，避免携带仅属于单一页面的业务判断。
- 请求层：统一组织 FastAPI API 调用、请求参数类型和响应类型。
- 共享逻辑层：承载多个页面重复使用的状态处理、格式化或业务交互流程。
- 上述职责边界是约束目标；具体目录名称和工程结构在构建方案确定后补充。

## 6. 开发与质量检查

- 新增或修改共享功能时，检查所有引用页面是否保持统一行为。
- 页面依赖 API 变更时，检查请求参数、响应展示、错误提示和空状态处理。
- 打包构建验证时，检查构建产物不包含写死的部署服务器接口主机地址；本地调试验证时，检查后端接口地址可直接配置。
- 涉及新增或修改 API 调用时，核对 FastAPI 契约、TypeScript 类型、空数据和失败状态是否一致更新。
- 涉及写操作或快速触发请求的交互时，验证不会无控制地产生重复请求。
- 涉及列表、详情或跨页面共享数据时，验证数据更新后的刷新表现保持一致。
- 涉及列表展示时，验证标题与列表内容是否居中且对齐一致。
- 涉及树形结构展示时，验证整行点击可以展开或收起节点，并验证刷新数据、编辑、修改、新增和删除后仍保持此前的展开或收起布局。
- 提交前运行项目实际配置的格式化、检查、测试或构建命令；相关工具和命令待项目确定后补充。
