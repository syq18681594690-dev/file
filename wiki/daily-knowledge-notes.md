# 日常知识积累

## 1. 适用范围

本文档用于沉淀日常开发过程中获得的通用知识、工具技巧、问题排查结论和可复用提醒。

属于明确专项规范的内容，应优先更新对应主题页面，例如 [Python 后端开发规范](python-backend-development-guide.md)、[Vue 3 前端开发规范](vue3-frontend-development-guide.md)、[前端开发经验](frontend-development-experience.md) 或 [跨页面功能一致性约定](cross-page-feature-consistency.md)。不适合单独建立主题页面、但值得长期保留的信息，可记录在本文档中。

## 2. 记录规则

- 每条知识应说明结论、适用场景和信息来源；涉及验证的内容应记录验证方式或验证状态。
- 可直接转化为强制执行要求的内容，应迁移或同步到对应规范页面，并在条目中建立链接。
- 尚未验证、仅作为尝试方向的信息，应明确标记为“待验证”，避免被误用为既定规则。
- 当同类知识条目持续增多或形成完整主题时，应拆分为独立 Wiki 页面，并在本页留下导航链接。

## 3. 条目模板

```markdown
## YYYY-MM-DD：主题

- 分类：工具 / 排查 / 实践 / 术语 / 待验证
- 结论：

- 来源：
- 验证状态：
- 关联页面：
```

## 4. 知识条目

## 2026-05-29：自定义 Wiki 知识库适合抽象为 Codex skill

- 分类：实践
- 结论：自定义 Wiki 知识库的初始化、目录生成、索引维护、日志记录、决策记录和 lint 流程适合做成 Codex skill；skill 应包含通用流程和模板，不应内置某个项目的具体业务规范内容。
- 适用场景：用户需要为新项目生成自己的知识库、开发规范 Wiki、团队规则库或长期维护型资料库。
- 来源：用户对话输入。
- 验证状态：待验证；当前项目已有手工维护流程，可作为 skill 的初始模板来源。
- 关联页面：[开发规范 Wiki 索引](index.md)、[维护日志](log.md)。

设计时应遵循以下边界：

- skill 负责生成 `raw/`、`wiki/`、索引、日志、进度和决策记录等骨架。
- skill 负责提供 ingest、query、lint、拆分主题、更新下次入口等操作流程。
- skill 可提供模板文件或脚本，提高新建知识库时的一致性。
- skill 不应写死当前项目的 Python、Vue、FastAPI 等业务规范；这些应由用户输入后生成到目标知识库。
- skill 的 `SKILL.md` 保持简洁，详细模板可放到 `assets/` 或 `references/` 中按需读取。
- 如果要真正落地，可命名为 `wiki-knowledge-base` 或 `project-wiki-maintainer`。

## 2026-05-29：Wiki 数据量增长后的更新同步防遗漏方法

- 分类：实践
- 结论：Wiki 数据量变大后，确实会增加遗漏、重复和规则冲突风险；应通过入口索引、主题页单一来源、维护日志、定期 lint 和下次会话入口来降低风险，而不是依赖记忆同步。
- 适用场景：新增规范、查询规范后沉淀结论、跨主题规则更新、长期维护多页面 Wiki。
- 来源：用户对话输入。
- 验证状态：已在当前 Wiki 工作流中部分采用，包括 `wiki/index.md`、`wiki/log.md`、`PROGRESS.md`、`DECISIONS.md` 和 lint 流程。
- 关联页面：[开发规范 Wiki 索引](index.md)、[维护日志](log.md)。

执行时遵循以下做法：

- 每次更新先从 [开发规范 Wiki 索引](index.md) 定位主题页，不直接凭印象改文件。
- 同一主题已有页面时更新原页面，避免创建重复规范页。
- 跨主题规则只保留一个主规则来源，其他页面用相对链接引用或补充实施要求。
- 每次 ingest、query 或 lint 后更新 [维护日志](log.md)，记录来源和结果，方便追溯。
- 会话结束前更新 `PROGRESS.md` 的“下次会话入口”，写明文件、行号和待完成事项。
- 数据量继续增长后，应定期执行 lint：检查失效链接、孤立页面、重复规则、冲突规则、过期结论和缺失来源。
- 当同类条目持续增多或一个主题需要多次引用时，再拆分为独立 Wiki 页面，并同步更新 [开发规范 Wiki 索引](index.md)。
