# Codex 开发规范 Wiki 维护指令

## 目录用途

- 本目录是由 Codex 维护的开发规范 Wiki，不默认作为可执行软件工程目录。
- 用户负责提供资料、要求和探索方向；Codex 负责归纳、交叉引用、更新页面与记录维护历史。
- 开始处理内容前，先读取 `wiki/index.md` 及相关页面，避免重复创建或产生冲突规则。

## 三层结构

- `raw/`：原始资料层。存放用户提供的来源资料或来源记录；已归档资料视为事实来源，不改写其内容。
- `wiki/`：知识整理层。由 Codex 创建和维护 Markdown 规范页面、综合结论、索引及日志。
- `AGENTS.md`：维护规则层。定义 Wiki 的目录约定和操作流程；工作流变化时更新本文件并记录决策。

## 文档整理原则

- 保留用户已明确的信息，例如版本、框架、命令、容量目标和操作触发条件。
- 修正明显笔误或标准名称时，在回复中说明修正，例如将 `requirem.txt` 整理为 `requirements.txt`。
- 对存在合理歧义的输入，先说明理解方式；如果错误理解会显著改变规范内容，则先询问用户。
- 将规则整理为可执行、可检查的表述，但不把用户未确定的版本、参数、目录结构、验收阈值或部署方案写成既定要求。
- 同一主题已有文档时更新原文档，避免无必要的新建重复文档。
- 只修改本次输入涉及的文档内容，不擅自创建项目代码、安装依赖、初始化工程或改动无关文件。
- Wiki 页面应使用相对链接关联相关页面；新建或移动页面时同步维护 `wiki/index.md`。
- 用户直接输入但未形成独立原始文件的内容，可以标注来源为“用户对话输入”；提供文件或需长期保留的外部资料时优先归档到 `raw/`。

## Wiki 页面约定

- 内容页面使用小写英文加连字符命名，例如 `python-backend-development-guide.md`。
- 内容页面按需要包含 YAML frontmatter：`title`、`type`、`status`、`updated` 与 `sources`。
- 规范页面正文按需要包含：适用范围、已确定约束、开发注意事项、执行或验证要求、待确认事项。
- 新增要求应归入对应主题章节；必要时更新汇总表或待确认事项，避免前后冲突。
- 用户明确取消或替换既有要求时，更新原规则，并在回复中说明被替换的内容。

## 操作流程

### Ingest：整理新输入或来源

1. 判断输入属于已有主题还是新主题，并阅读 `wiki/index.md` 与相关页面。
2. 如用户提供需归档的原始文件或外部资料，将来源保存或记录到 `raw/`；不要改写已有来源。
3. 提取明确约束、补充信息、冲突点及待确认项，更新或创建 `wiki/` 中的主题页面。
4. 更新 `wiki/index.md` 的页面目录和更新时间。
5. 在 `wiki/log.md` 追加 `## [YYYY-MM-DD] ingest | 标题` 记录，写明来源与更新结果。

### Query：基于 Wiki 回答问题

1. 从 `wiki/index.md` 定位相关页面，再读取必要内容。
2. 回答时引用使用到的 Wiki 页面；信息不足时明确指出资料缺口。
3. 如果回答形成了可复用的新规范、比较或结论，将其沉淀到 `wiki/` 并追加 `query` 日志。

### Lint：检查 Wiki 健康度

1. 检查失效链接、孤立页面、重复或冲突规则、过期结论和缺失来源。
2. 对可确定的问题直接修复；涉及用户决定的冲突或缺口，列为待确认事项。
3. 在 `wiki/log.md` 追加 `## [YYYY-MM-DD] lint | 标题` 记录。

## 当前导航

- `wiki/index.md`：Wiki 总索引，所有查询与整理工作的入口。
- `wiki/log.md`：按时间追加的维护历史。
- `wiki/python-backend-development-guide.md`：Python 后端技术环境、接口框架、数据访问、服务启动、依赖管理、并发目标及 Git 提交约定。
- `wiki/frontend-development-experience.md`：前端技术选型、实现约定、接口联调及质量交付经验入口。
- `wiki/vue3-frontend-development-guide.md`：Vue 3 前端组件、状态、接口联调和质量检查专项规范。
- `wiki/cross-page-feature-consistency.md`：前后端多页面复用同一功能时的一致性规则与例外告知要求。
- `wiki/daily-knowledge-notes.md`：尚未归入专项规范的日常知识、工具技巧与排查结论积累入口。

## Git 提交与推送

- 未收到用户明确的提交指令时，不执行 `git commit`。
- 当用户明确要求“提交 git”或表达同等提交意图时，检查本次文档变更后执行 `git commit`。
- 如果当前目录不是可提交的 Git 仓库，说明无法提交的原因，不擅自初始化仓库。
- 禁止自动执行 `git push`。
- 只有用户明确要求推送到远程仓库时，才可执行 `git push`；提交指令不等同于推送授权。

## 文件自动维护规则

- 如果项目存在 `PROGRESS.md`，完成任何文档整理子任务后立即更新状态，并在会话结束前更新“下次会话入口”（文件路径、行号、待完成事项）。
- 如果项目存在 `DECISIONS.md`，选择文档组织方案、分类方式或维护策略时，按 `## 日期：决策标题 / 原因 / 否决方案` 格式追加记录。
- 如果项目存在 `CHANGELOG.md`，每次执行 `git commit` 后在文件顶部追加本次变更摘要，按新增、修复、重构三类记录。
