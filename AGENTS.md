# Codex 开发笔记维护指令

## 目录用途

- 本目录用于维护用户的开发笔记和开发规范文档，不默认作为可执行软件工程目录。
- 用户输入技术选型、开发注意事项、流程约定或补充要求时，将其整理、优化并写入对应的规范文档。
- 已有后端规范文档为 `BACKEND_DEVELOPMENT_GUIDE.md`；与后端开发相关的新输入优先更新该文件。

## 文档整理原则

- 保留用户已明确的信息，例如版本、框架、命令、容量目标和操作触发条件。
- 修正明显笔误或标准名称时，在回复中说明修正，例如将 `requirem.txt` 整理为 `requirements.txt`。
- 对存在合理歧义的输入，先说明理解方式；如果错误理解会显著改变规范内容，则先询问用户。
- 将规则整理为可执行、可检查的表述，但不把用户未确定的版本、参数、目录结构、验收阈值或部署方案写成既定要求。
- 同一主题已有文档时更新原文档，避免无必要的新建重复文档。
- 只修改本次输入涉及的文档内容，不擅自创建项目代码、安装依赖、初始化工程或改动无关文件。

## 推荐文档结构

- 规范文档按需要包含：适用范围、已确定约束、开发注意事项、执行或验证要求、待确认事项。
- 新增要求应归入对应主题章节；必要时更新汇总表或待确认事项，避免前后冲突。
- 用户明确取消或替换既有要求时，更新原规则，并在回复中说明被替换的内容。

## 当前已维护文档

- `BACKEND_DEVELOPMENT_GUIDE.md`：后端技术环境、接口框架、服务启动、依赖管理、并发目标及 Git 提交约定。

## Git 提交

- 未收到用户明确的提交指令时，不执行 `git commit`。
- 当用户明确要求“提交 git”或表达同等提交意图时，检查本次文档变更后执行 `git commit`。
- 如果当前目录不是可提交的 Git 仓库，说明无法提交的原因，不擅自初始化仓库。

## 文件自动维护规则

- 如果项目存在 `PROGRESS.md`，完成任何文档整理子任务后立即更新状态，并在会话结束前更新“下次会话入口”（文件路径、行号、待完成事项）。
- 如果项目存在 `DECISIONS.md`，选择文档组织方案、分类方式或维护策略时，按 `## 日期：决策标题 / 原因 / 否决方案` 格式追加记录。
- 如果项目存在 `CHANGELOG.md`，每次执行 `git commit` 后在文件顶部追加本次变更摘要，按新增、修复、重构三类记录。
